# Module 04: Compression - 上下文压缩

## 为什么需要压缩

AI 的上下文窗口有限。对话越长，最早的内容越可能被丢弃。
压缩机制在保留关键信息的同时，减少 Token 消耗。

## 压缩触发条件

| 条件 | 行为 |
|------|------|
| 对话超过 50 轮 | 标记最早 20 轮为"待压缩" |
| 对话超过 100 轮 | 再压缩 20 轮 |
| 每增加 50 轮 | 再压缩 20 轮 |
| 手动触发 | /压缩记忆 |

## 压缩规则

- 保留最近 30 轮原文不被压缩
- 只压缩早于最近 30 轮的内容
- 每次压缩 20 轮为一个单位
- 压缩结果写入 archive.jsonl

## 压缩格式

```json
{
  "date": "2024-06-07",
  "summary": "压缩后的对话摘要",
  "tags": ["标记"],
  "tokens_saved": 2800,
  "importance": 3
}
```

- summary: 压缩后的摘要文本
- tags: 对话涉及的主题标签
- tokens_saved: 节省的 Token 估算
- importance: 重要性（1-5），用于后续保留决策

## Token 节省估算

| 对话轮次 | 原始 Token | 压缩后 | 节省 |
|---------|-----------|--------|------|
| 50 轮 | ~15,000 | ~12,000 | ~20% |
| 100 轮 | ~30,000 | ~18,000 | ~40% |
| 200 轮 | ~60,000 | ~25,000 | ~58% |

## 保留策略

- 重要性 >= 4 的记录永久保留
- 重要性 < 3 且超过 30 天的记录可清理
- 手动 /清理记忆 可触发清理
