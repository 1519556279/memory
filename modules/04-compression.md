# Module 04: Compression — 上下文压缩

## 为什么需要压缩

AI 的上下文窗口有限。典型情况：

| 对话长度 | 问题 | 结果 |
|---------|------|------|
| 0-30 轮 | 正常 | 完整上下文 |
| 30-50 轮 | 开始吃紧 | 最早的内容可能被截断 |
| 50-100 轮 | 严重 | 早期设定/偏好丢失 |
| 100+ 轮 | 非常严重 | 只剩最近几轮 |

**压缩机制在保留关键信息的同时，减少 Token 消耗。**

## 压缩触发条件

| 条件 | 行为 |
|------|------|
| 对话超过 50 轮 | 自动压缩最早的 20 轮 |
| 对话超过 100 轮 | 再压缩 20 轮（累计 40 轮压缩） |
| 每增加 50 轮 | 再压缩 20 轮 |
| 用户输入 `/压缩记忆` | 立即压缩 |

## 压缩规则（黄金法则）

| 规则 | 说明 |
|------|------|
| 保留最近 30 轮原文 | 永远不压缩最近的 30 轮 |
| 每次压缩 20 轮 | 一批一批地压缩 |
| 只压缩一次 | 已压缩的内容不再重复压缩 |
| 摘要写入硬盘 | 存入 archive.jsonl，下次对话可读 |

## 压缩格式

```json
{
  "date": "2024-06-07",
  "summary": "讨论了记忆技能的重构方案，确定拆分为 6 个模块。用户偏好：TypeScript、2空格。删除了 computer-control。",
  "tags": ["重构", "记忆"],
  "tokens_saved": 3200,
  "importance": 4
}
```

## 压缩摘要撰写规范

写摘要时，必须包含以下信息（如果对话中有的话）：

1. ✅ 用户明确表达的偏好
2. ✅ 做出的关键决定
3. ✅ 项目重要进展
4. ✅ 安装/卸载的操作
5. ✅ 未完成的任务/待办
6. ❌ 不包含闲聊内容
7. ❌ 不包含技术细节（除非是关键决策）

## Token 节省计算

| 对话轮次 | 未压缩 Token | 压缩后 Token | 节省 |
|---------|-------------|-------------|------|
| 50 轮 | ~15,000 | ~12,000 | ~20% |
| 100 轮 | ~30,000 | ~18,000 | ~40% |
| 150 轮 | ~45,000 | ~22,000 | ~51% |
| 200 轮 | ~60,000 | ~25,000 | ~58% |

计算公式：`节省率 = 1 - (压缩后 / 未压缩)`

## 压缩后的上下文结构

```
[最近 30 轮原文]
↕
[20 轮摘要 #1] ← 最早
[20 轮摘要 #2]
[20 轮摘要 #3]
```

## 存档保留策略

| 重要性 | 保留时间 | 说明 |
|--------|---------|------|
| 5（关键） | 永久保留 | 重要决策、里程碑 |
| 4（重要） | 永久保留 | 明确偏好、项目进度 |
| 3（普通） | 90 天 | 一般讨论 |
| 2（低） | 30 天 | 闲聊、临时信息 |
| 1（可丢弃） | 7 天 | 无价值内容，可随时删除 |

## 手动触发

用户说 `/压缩记忆` 时：

1. 检查当前对话轮次
2. 如果有可压缩的旧轮次（> 30 轮前的内容）
3. 生成摘要
4. 写入 archive.jsonl
5. 输出压缩报告：`📦 已压缩 XX 轮，节约 ~XXX Tokens`
